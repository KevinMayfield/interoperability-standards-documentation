Note to Implementers: FHIR Capabilities

Any FHIR capability may be 'allowed' by the system unless explicitly marked as "SHALL NOT". A few items are marked as MAY in the Implementation Guide to highlight their potential relevance to the use case.

Summary

The summary table lists the resources that are part of this configuration, and for each resource it lists:

  • The relevant profiles (if any)
  • The interactions supported by each resource (Read, Search, Update, and Create, are always shown, while VRead, Patch, Delete, History on Instance, or History on Type are only present if at least one of the resources has support for them.
  • The required, recommended, and some optional search parameters (if any).
  • The linked resources enabled for _include
  • The other resources enabled for _revinclude
  • The operations on the resource (if any)

  • Personal Demographics Service FHIR API

Used by

  • EPS consumers.

SHOULD and SHALL conformance is taken from UK Core FHIR Access - MAY is used to indicate no conformance (added due to rendering issue in the publisher)

Search Parameters
ConformanceParameterTypeDocumentation
SHALLidentifiertoken

A search by NHS Number. Missing from PDS is a read operation instead - so using transient id instead of identifier

SHALLgivenstring
SHOULDfamilystring
SHOULDgenderstring
SHOULDbirthdatedate
MAY_fuzzy-matchstring

PDS using non standard parameter

MAY_exact-matchstring

PDS is using non standard parameter. Do not believe this is :exact from modifiers although it sounds similar and may cause confusion

MAY_historystring

PDS is using non standard parameter. Do not believe this is _history from history this probably results in confusion for developers

MAY_max-resultsstring

PDS is using non standard parameter. PDS probably should be using _count _count which should be ok as PDS does not support paging

MAYdeath-datedate
MAYaddress-postalcodestring
MAYaddress-postcodestring

PDS is using non standard parameter - will be deprecated in the future

MAYgeneral-practitionerreference

A search by ODS Code. PDS SHOULD be using general-practitioner:identifier with token type to follow NHS Digital FHIR policy. It is workable as it is but this is searching on an id which is transient (changeable), searching on identifier is recommended.

MAYemailtoken
MAYphonetoken
